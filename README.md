## สเปคของ ADT

### ค่านามธรรม (A)

ลำดับของเพลง เช่น `[เพลงA, เพลงB, เพลงC]` โดยลำดับมีความหมาย — เพลงแรกในรายการคือเพลงที่เล่นก่อน

### Representation (R)

```java
private final List<String> songs;
```

### กฎที่เพลย์ลิสต์ต้องรักษาไว้เสมอ

- ต้องมีรายการเพลงอยู่จริง (ไม่เป็น `null`)
- ไม่มีเพลงใดเป็น `null`
- ไม่มีชื่อเพลงที่เป็นสตริงว่าง
- **ชื่อเพลงห้ามซ้ำกัน**
- มีได้ไม่เกิน `MAX_SONGS` (100) เพลง

## งานที่ต้องทำ

### งานที่ 1 — เขียน AF, RI และ Safety from rep exposure (TODO 1–3)

เขียนเป็นคอมเมนต์ไว้เหนือ `private final List<String> songs;`

**AF** ต้องบอกว่า `songs` **หมายถึงอะไร** ไม่ใช่บอกว่ามันเป็นตัวแปรชนิดอะไร

> ตัวอย่างที่ผิด: `AF(songs) = ตัวแปร List ที่เก็บ String`
> ตัวอย่างที่ถูก: `AF(songs) = เพลย์ลิสต์ที่เล่นเพลง songs.get(0), songs.get(1), ... ตามลำดับ`

**RI** ต้องเขียนให้ครบทั้ง 5 ข้อตามหัวข้อ "กฎที่เพลย์ลิสต์ต้องรักษาไว้เสมอ" ด้านบน

**Safety from rep exposure** ให้อธิบายว่าคุณป้องกันการรั่วด้วยวิธีใดบ้าง

### งานที่ 2 — เขียน `checkRep()` (TODO 4)

แปลง RI ทุกข้อให้เป็น `assert` หนึ่งบรรทัด พร้อมข้อความอธิบายหลังเครื่องหมาย `:`

```java
assert songs != null : "songs ต้องไม่เป็น null";
```

**คำใบ้สำหรับข้อ "ชื่อเพลงไม่ซ้ำ":** ใช้ `Set` ช่วยจำว่าเจอเพลงไหนไปแล้วบ้าง เมธอด `Set.add()` คืน `false` เมื่อสมาชิกมีอยู่แล้ว

```java
Set<String> seen = new HashSet<>();
for (String s : songs) {
    assert seen.add(s) : "ชื่อเพลงซ้ำ: " + s;
}
```

### งานที่ 3 — เขียน Creator ตัวที่สอง (TODO 5)

`Playlist(List<String> initial)` สร้างเพลย์ลิสต์จากรายชื่อที่ให้มา

- ถ้า `initial` ผิดเงื่อนไข (เป็น `null`, มีเพลงซ้ำ, มีเพลงเป็น `null` หรือสตริงว่าง, เกิน `MAX_SONGS`) ให้ `throw new IllegalArgumentException` พร้อมข้อความอธิบาย
- **ห้ามเก็บ reference ของ `initial` ตรง ๆ** ต้องคัดลอกเข้ามาใหม่
- เรียก `checkRep()` ก่อนจบเมธอด

> **ทำไมต้อง throw ไม่ใช่ assert?**
> `initial` มาจาก client ถ้าเขาส่งค่าผิดคือ**ความผิดของ client** ต้องใช้ exception
> ส่วน `assert` ใช้ตรวจ rep ของเราเอง ซึ่งถ้าพังแปลว่า**เราเขียนบั๊ก**
> อีกเหตุผลคือ `assert` ถูกปิดอัตโนมัติเมื่อไม่ใส่ `-ea` ตอนรันจริงจะไม่มีการตรวจเลย

### งานที่ 4 — เขียน Mutators (TODO 6–7)

| เมธอด | พฤติกรรม |
|---|---|
| `add(String song)` | เพิ่มต่อท้าย · คืน `false` ถ้ามีเพลงนี้แล้วหรือเต็มแล้ว · `throw` ถ้า `song` เป็น `null` หรือสตริงว่าง |
| `remove(String song)` | ลบเพลง · คืน `true` ถ้าลบสำเร็จ, `false` ถ้าไม่พบ |

อย่าลืมเรียก `checkRep()` ก่อน `return` ทุกเส้นทางที่แก้ไข rep

### งานที่ 5 — เขียน Observers (TODO 8–10)

| เมธอด | พฤติกรรม |
|---|---|
| `size()` | จำนวนเพลง |
| `contains(String song)` | มีเพลงนี้หรือไม่ |
| `songs()` | รายชื่อเพลงทั้งหมดตามลำดับ |

**จุดสำคัญที่สุดของแลปนี้อยู่ที่ `songs()`** — ถ้าคืน `songs` ตรง ๆ client จะแก้ rep ข้างในได้ทันที

```java
List<String> got = p.songs();
got.clear();          // ถ้า songs() คืน reference ตรง ๆ เพลย์ลิสต์จะว่างไปด้วย!
```

Observer ไม่ต้องเรียก `checkRep()` เพราะไม่แตะ rep

### งานที่ 6 — เขียน Producer (TODO 11)

`shuffled()` คืนเพลย์ลิสต์**ตัวใหม่**ที่มีเพลงชุดเดียวกันแต่สลับลำดับ

- **ห้ามแก้เพลย์ลิสต์เดิม** (`this`) เด็ดขาด — นี่คือหัวใจของคำว่า producer
- ใช้ `Collections.shuffle(list)` ช่วยได้ แต่ต้องสลับบน list ที่คัดลอกมาแล้วเท่านั้น

## ข้อผิดพลาดที่พบบ่อย

- **เขียน AF เป็นการอธิบาย field** — AF ต้องบอกความหมายในโลกจริง ไม่ใช่บอกว่า `songs` คือตัวแปร List
- **สับสน AF กับ RI** — AF ตอบ "หมายถึงอะไร" ส่วน RI ตอบ "ถูกกฎหรือไม่"
- **เขียน `checkRep()` แล้วไม่เคยเรียก** — เขียนไว้สวยงามแต่ไม่เรียก เท่ากับไม่มีประโยชน์
- **คัดลอกแค่ขาออก ลืมขาเข้า** — constructor ที่รับ `List` ก็ต้องคัดลอกเหมือนกัน
- **ใช้ `assert` ตรวจ input ของ client** — ต้องใช้ exception เพราะ assert ถูกปิดตอนรันจริง
- **ลืมใส่ `-ea` ตอนรัน** — `checkRep()` จะไม่ทำงาน เทสต์อาจผ่านทั้งที่โค้ดพัง
